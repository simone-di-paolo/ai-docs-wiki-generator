This repository contains the source code for React, a JavaScript library for building user interfaces, along with its optimizing compiler, various renderer implementations, and comprehensive development infrastructure. It supports declarative UI development across web, native, and server environments.

The React Compiler transforms source code during the build step, minimizing re-renders and enforcing core React rules. An interactive playground demonstrates the compiler's behavior and transformations. For a deeper understanding, see React Compiler.

Feature Flag Management

zoom_in

React utilizes a system of feature flags to control the availability of experimental features, modify runtime behavior, and facilitate A/B testing across different environments. This allows for conditional compilation and execution, enabling developers to test new features or specific platform behaviors without affecting all users.

Feature flags are primarily defined in JavaScript files that specify their state for various environments. For instance, packages/shared/ReactFeatureFlags.js contains default (canary) flags, while environment-specific forks like packages/shared/forks/ReactFeatureFlags.www.js for the web and packages/shared/forks/ReactFeatureFlags.native-fb.js for React Native provide overrides. A special __VARIANT__ value in these fork files indicates that a flag should be tested in both true and false states, simulating A/B testing scenarios.

During development and testing, specific pragmas and functions are used to interact with feature flags. The @gate pragma, found in test files, allows an entire test to be skipped if a specified flag is false, indicating that the feature is completely unavailable without that flag. For scenarios where behavior differs based on a flag but the feature always exists, an inline gate() function is used within tests to execute different code paths.

To inspect and compare the states of feature flags across various release channels, a command-line utility is available. This tool can display all flags, compare flag values between two specified channels using the --diff option, group flags by their cleanup status, and output results in CSV format. Supported channels include www, canary, next, experimental, rn, and rn-fb. The utility provides a legend to interpret flag states, indicating whether a flag is enabled (), disabled (), a __VARIANT__ (), or profiling-only (). This process is detailed in Feature Flag Management. The workflow for managing feature flags, including adding new ones and debugging channel-specific issues, is outlined in .claude/skills/feature-flags/SKILL.md. For a broader view of how these flags integrate into overall project workflows, refer to React Project Management and Contribution.

Error Code Management

zoom_in

React manages its error codes to enhance debugging in production environments while minimizing bundle size. Instead of shipping verbose error messages to clients, the system replaces them with compact numerical IDs. This allows for smaller production builds and obfuscates internal error details, while still providing a mechanism to map these IDs back to their original messages for developers.

The process involves maintaining a central mapping of error IDs to messages in a generated file, codes.json. This file is updated by a Node.js script, extract-errors.js, which scans the codebase (specifically build artifacts) for unminified error messages and assigns unique numerical codes to any new ones it finds. Developers can initiate this process using yarn extract-errors. The script ensures that error codes are unique and sequential, and it prevents duplicate messages from being assigned new codes. Further details on how this process works can be found in .claude/skills/extract-errors/SKILL.md.

During the build process, a Babel plugin, implemented in transform-error-messages.js, plays a crucial role. This plugin modifies calls to the Error constructor, replacing descriptive string messages with their corresponding numerical error codes in production builds. If an error message does not have an assigned code, the plugin inserts a FIXME comment and an <expected-error-format> comment, helping identify untransformed errors without breaking the build. The plugin also handles various string interpolation methods and preserves additional arguments passed to the Error constructor.

To inspect and compare feature flag states across different release channels, which can influence how errors are generated or surfaced, developers can use the flags command-line utility. This utility, detailed in Feature Flag Management and .claude/skills/flags/SKILL.md, helps in understanding environment-specific behaviors. The overall development process, including code quality, testing, and contribution guidelines, is described in React Project Management and Contribution.

The process is orchestrated by scripts within the scripts/flow directory, which dynamically create .flowconfig files to ensure that Flow's static analysis correctly accounts for the unique characteristics of different React renderers, such as DOM, Native, Server, and Flight.

Configuration generation is handled by scripts/flow/createFlowConfigs.js. This script identifies and resolves renderer-specific "fork" files, which are alternative implementations for shared interfaces, and then incorporates these into a tailored .flowconfig. This customization includes determining ignored paths and setting module.name_mapper directives to guide Flow's understanding of the codebase. The configurations are built upon a configTemplate, allowing for consistent base settings augmented with dynamic renderer-specific additions.

Execution of Flow type checks is managed by scripts/flow/runFlow.js. This script ensures that the correct renderer's .flowconfig is activated by copying it to the project root before Flow is run, thereby enabling consistent type checking across various build environments and editor integrations.

Beyond configuration, the system provides extensive type declarations for global variables, browser APIs, and internal modules. For instance, scripts/flow/environment.js declares global constants like __PROFILE__ and __EXPERIMENTAL__, extends browser types for console, and redefines React$Element to expose internal properties. Similarly, scripts/flow/react-devtools.js specifically declares global Flow types for the React DevTools build environment, including flags such as __EXTENSION__ and __IS_CHROME__. For React Native, scripts/flow/react-native-host-hooks.js defines critical types for its renderer implementations, including callbacks and internal types for ReactNativePrivateInterface and nativeFabricUIManager.

The project also maintains a flow-typed directory, which serves as a central repository for Flow type definitions. This includes flow-typed/environments, which provides comprehensive types for various JavaScript execution environments, such as the Browser Object Model (BOM), CSS Object Model (CSSOM), Document Object Model (DOM), WebGL, W3C Geometry Interfaces, HTML elements, Service Workers, Web Streams, and Web Animations. These definitions ensure that Flow can perform accurate static analysis on code interacting with these native APIs. Additionally, flow-typed/npm contains type definitions for external npm packages like error-stack-parser and minimist, further enhancing type safety for third-party dependencies.

While $FlowFixMe suppressions are available, their use is discouraged and subject to scrutiny, with a strong emphasis on providing correct type imports. The overall goal is to achieve comprehensive static analysis, catching type-related errors at compile time and improving code reliability and maintainability. React Compiler

zoom_in

The React Compiler is a performance optimization tool designed to minimize re-renders in React applications and enforce adherence to the "Rules of React." Its primary goal is to achieve predictably fast performance and neutral startup times, allowing developers to write idiomatic React code without manually implementing memoization APIs like React.memo(), useMemo(), or useCallback(). The compiler achieves this by transforming source code during the build step, leveraging various analyses and optimizations to generate optimized JavaScript.

The core of the React Compiler is implemented as a Babel plugin, located in compiler/packages/babel-plugin-react-compiler. This plugin identifies functions for compilation, converts Babel Abstract Syntax Trees (ASTs) into a High-level Intermediate Representation (HIR), and then applies a multi-stage compilation pipeline. This pipeline includes steps such as Static Single Assignment (SSA) form conversion, type inference, mutation analysis, and the construction of reactive scopes. These scopes group related values and instructions, allowing the compiler to determine when re-computation is necessary. The optimized HIR is then converted back into a Babel AST, which replaces the original code.

To integrate compiler diagnostics into the development workflow, the React Compiler provides an ESLint plugin (compiler/packages/eslint-plugin-react-compiler). This plugin leverages the same core compiler logic to detect and report rule violations and potential issues within React code, often providing autofix capabilities.

For enforcing immutability during development and testing, a dedicated utility package (compiler/packages/make-read-only-util) is available. This tool helps track and report mutation violations, which is crucial for maintaining predictable state management in React applications.

The compiler's runtime library, found in compiler/packages/react-compiler-runtime, provides essential utilities for compiled React applications, including a backwards-compatible shim for React Compiler runtime APIs. This ensures that applications compiled with the React Compiler can run across different React versions, offering features such as memoization caches and stricter hook usage validation.

Developers can use a VS Code extension, located in compiler/packages/react-forgive, to enhance their React development experience. This extension provides a language server for code analysis, visualizes useEffect dependencies, and integrates code optimization features.

An interactive web-based playground, housed in compiler/apps/playground, allows users to experiment with the React Compiler. This environment enables code editing, configuration adjustments, and the visualization of compiled JavaScript, source maps, and intermediate representations. The playground supports end-to-end testing and provides detailed error reporting, offering a practical way to understand how the compiler transforms code. For more details, see React Compiler Playground.

The design goals, architecture, and development workflow of the React Compiler are thoroughly documented in compiler/docs. This documentation outlines the compiler's purpose, its internal mechanisms, and guidelines for contributing to its development. The project also includes a command-line interface (CLI) tool (compiler/packages/snap) that facilitates compiler development and testing by offering functionalities for test execution, code minimization, and compilation. Debugging capabilities, including automated error investigation agents, are also available for efficiently identifying and resolving compiler issues. See Debugging React Compiler with Automated Tools.

React Compiler Internals: Compilation Pipeline and IRs

zoom_in

The React Compiler transforms React applications to minimize re-renders and enforce the "Rules of React" through a multi-stage compilation pipeline. This process involves converting source code through various Intermediate Representations (IRs) to apply optimizations and generate more efficient JavaScript. The core of this process is implemented within the Babel plugin, primarily orchestrated by functions like compileProgram and compileFn located in compiler/packages/babel-plugin-react-compiler/src/Entrypoint.

The compilation typically begins by identifying functions within the source code that are candidates for optimization. These functions are then processed through a series of passes, each designed to perform specific analyses and transformations. The sequence of these passes is crucial, with each stage building upon the output of the preceding one. Detailed documentation for each pass can be found in the compiler/packages/babel-plugin-react-compiler/docs directory.

Intermediate Representations (IRs)

Two primary IRs are central to the React Compiler's pipeline:

  • High-level Intermediate Representation (HIR): The initial transformation converts the Babel Abstract Syntax Tree (AST) into an HIR, which takes the form of a Control-Flow Graph (CFG). This representation explicitly defines the control flow within a function using basic blocks, instructions, and terminals, and assigns unique identifiers to variables. This conversion is handled by passes such as lowerStatement and lowerExpression during the lowering phase. Concepts related to HIR, such as HIRFunction, Instruction, Terminal, Place, and Phi nodes, are defined in compiler/CLAUDE.md.

  • ReactiveFunction: Following extensive analysis and optimization in HIR, the compiler transforms the HIR into a tree-based ReactiveFunction. This IR restores structured control flow and eliminates Phi nodes, serving as an optimized representation closer to the final JavaScript output. This transformation occurs during the buildReactiveFunction pass.

Key Compilation Passes

The compilation pipeline involves numerous passes, each contributing to the optimization process:

  • HIR Lowering (01-lower.md): This initial pass converts the Babel AST of a function into an HIRFunction CFG, establishing explicit control flow and unique identifiers.
  • SSA Form Conversion (02-enterSSA.md, 03-eliminateRedundantPhi.md): The HIR is transformed into Static Single Assignment (SSA) form, where each variable is assigned exactly once. This involves renaming variables and inserting Phi nodes to merge values from different control flow paths. Redundant Phi nodes are subsequently removed.
  • Constant Propagation (04-constantPropagation.md): This optimization pass identifies and replaces variables with their constant values, and can also eliminate unreachable code.
  • Dead Code Elimination (05-deadCodeElimination.md): This pass removes code that does not affect the program's output, using a mark-and-sweep algorithm.
  • Type Inference (06-inferTypes.md): The compiler infers types for all identifiers within the HIR, categorizing them as primitives, objects, functions, etc., using a constraint-based algorithm.
  • Mutation and Aliasing Analysis (07-analyseFunctions.md, 08-inferMutationAliasingEffects.md, 09-inferMutationAliasingRanges.md): These passes analyze how data flows and changes within the function. They infer aliasingEffects for nested functions and determine mutation effects (e.g., Create, Mutate, Freeze) and mutableRange for identifiers using abstract interpretation. This system helps track how variables are modified and whether they can be aliased, impacting memoization decisions. More information on the AliasingEffects system can be found in compiler/CLAUDE.md.
  • Reactive Scope Construction (10-inferReactivePlaces.md, 11-inferReactiveScopeVariables.md, 13-alignMethodCallScopes.md, 14-alignObjectMethodScopes.md, 15-alignReactiveScopesToBlockScopesHIR.md, 16-mergeOverlappingReactiveScopesHIR): These passes identify reactive parts of the code and group co-mutating variables into ReactiveScopes. This involves determining "reactive places" and then refining these scopes by aligning them with control flow boundaries and merging overlapping scopes.
  • Reactive Function Building (21-buildReactiveFunction.md): After various analyses and transformations in HIR, this pass converts the optimized HIR into a tree-based ReactiveFunction, preparing it for code generation.

Other significant optimization passes include instructionReordering, inlineJsxTransform, lowerContextAccess, optimizeForSSR, outlineFunctions, and outlineJSX, all located in compiler/packages/babel-plugin-react-compiler/src/Optimization. The compiler also supports dropManualMemoization to remove explicit useMemo/useCallback calls and inferEffectDependencies for automatic useEffect dependency inference.

Finally, a Codegen step converts the ReactiveFunction back into a Babel AST node, which then replaces the original node in the source code. The compiler integrates with ESLint to report rule violations, leveraging the same core logic. See React Compiler for details on the ESLint integration.

Debugging React Compiler with Automated Tools

zoom_in

The React Compiler utilizes an automated agent to investigate errors and pinpoint their root causes, thereby streamlining the debugging process. This agent's methodology is detailed in compiler/.claude/agents/investigate-error.md and automates several steps to diagnose compiler issues.

The procedure commences with the creation of a new test fixture file, which contains the code exhibiting the problematic behavior. This fixture then undergoes a debug compilation using the yarn snap command, incorporating the -d flag. This flag is critical as it produces detailed output illustrating the program's state after each compilation pass, enabling an in-depth understanding of how the code transforms throughout the compiler's pipeline.

Should a compilation yield an error, the agent employs a minimization technique to simplify the failing test case to its most basic form. This is accomplished through yarn snap minimize --update <path-to-fixture>, which iteratively prunes the code while preserving the error, aiming for the most minimal reproduction of the bug. Following this automated minimization, manual refinements can further isolate the specific language feature or pattern causing the issue.

A central element of the investigation involves comparing the debug outputs from both the minimal failing fixture and a corresponding non-failing version. By examining the intermediate representations and effects pass-by-pass, the agent can identify the precise stage where the compiler's behavior diverges from expectations. This allows for a targeted investigation into the specific compiler logic. Relevant directories for such investigations include /facebook/react/compiler/packages/babel-plugin-react-compiler/src/HIR/ for Intermediate Representation definitions, /facebook/react/compiler/packages/babel-plugin-react-compiler/src/Inference/ for effect inference, /facebook/react/compiler/packages/babel-plugin-react-compiler/src/Validation/ for validation passes, /facebook/react/compiler/packages/babel-plugin-react-compiler/src/Optimization/ for optimization passes, and /facebook/react/compiler/packages/babel-plugin-react-compiler/src/ReactiveScopes/ for reactive scope analysis. The order in which these passes execute is defined within the Babel plugin.

The automated investigation culminates in a structured report. This report encompasses a description of the bug, the minimal failing and non-failing fixtures, identification of the problematic compiler pass, an analysis of the root cause, precise references to suspect code locations, and suggested directions for potential fixes. This structured approach aims to streamline the debugging workflow for React Compiler developers, as further explained in React Compiler Internals: Compilation Pipeline and IRs.

React Renderer Implementations

The React ecosystem provides a variety of rendering mechanisms tailored for different environments, encompassing both client-side and server-side rendering for web applications, as well as native mobile platforms. These renderers are responsible for translating React components into host-specific UI elements, managing their lifecycle, handling events, and optimizing updates.

At its core, the React DOM (packages/react-dom) facilitates rendering React components into the browser's Document Object Model (DOM) and offers server-side rendering (SSR) capabilities. For client-side applications, createRoot and hydrateRoot handle mounting and hydrating React component trees in the browser, managing their lifecycle from creation to unmounting. React DOM also includes utilities for resource preloading, such as preload and preinit, and provides mechanisms for form state management. For server environments, React DOM offers both modern streaming SSR APIs like renderToPipeableStream and renderToReadableStream (often referred to as "Fizz") and legacy synchronous methods such as renderToString. These server-side implementations generate HTML from React components, enabling faster initial page loads and improved user experience. Additionally, static prerendering functionalities like prerender and resumeAndPrerender are available for generating static HTML at build time across various environments including Node.js, browsers, and Edge runtimes.

For mobile applications, the React Native renderer (packages/react-native-renderer) bridges React components to native UI views. It supports both the legacy (Paper) and newer (Fabric) architectures, defining how React components map to and interact with native views. This involves managing component mounting, updates, and unmounting, as well as providing configurations for the reconciler to interact with platform-specific UI managers. A key aspect is its sophisticated event system, which translates native events into React's synthetic event model, handling touch and gesture events through mechanisms like the ResponderEventPlugin. Error handling is also integrated to provide platform-specific error reporting.

Beyond traditional rendering, React supports advanced paradigms like React Server Components (RSC). The packages/react-server package provides experimental server-side runtimes for RSC using "Flight" and "Fizz". Flight enables components to execute exclusively on the server, serializing their output for the client. This involves managing client and server references, allowing for the transfer of functions and modules between the client and server environments. The integration with Webpack, managed by packages/react-server-dom-webpack, facilitates the bundling and communication required for RSC. This includes Node.js ESM loader hooks for transforming "use client" and "use server" directives, and a Webpack plugin for discovering client modules and generating manifests. These mechanisms ensure that the correct client-side bundles are identified and loaded, enabling seamless interaction between server-rendered components and client-side interactivity.

React DOM: Client-side Rendering

zoom_in

React DOM's client-side rendering capabilities enable the interactive display and management of React component trees within a web browser's Document Object Model (DOM). This process begins with the creation or hydration of a root, which serves as the entry point for React to take control of a part of the DOM.

The createRoot function, exposed via packages/react-dom/client.js, is used to mount a React application into an empty DOM node, initiating a new concurrent rendering root. This function returns an object with render and unmount methods, allowing for the dynamic lifecycle management of the React tree within the browser. For applications that are pre-rendered on a server, hydrateRoot (also from packages/react-dom/client.js) is used instead. It attaches React to existing server-generated HTML, making it interactive without discarding and re-rendering the initial markup.

React components, once mounted, undergo a lifecycle of updates driven by state changes, prop changes, or forced re-renders. These updates are reconciled by React's internal mechanisms, which efficiently identify and apply necessary changes to the browser's DOM. This ensures that the user interface remains synchronized with the application's state.

For internal Facebook applications, specialized integrations exist to manage error handling and root behavior. For example, createRoot and hydrateRoot in packages/react-dom/src/client/ReactDOMRootFB.js are wrapped to integrate with custom error reporting mechanisms, such as wwwOnUncaughtError and wwwOnCaughtError. These mechanisms can trigger custom error dialogs via ReactFiberErrorDialogWWW.showErrorDialog, as mocked in packages/react-dom/src/client/__mocks__/ReactFiberErrorDialog.js. This allows for a consistent and managed approach to displaying and logging runtime errors in production environments.

Resource preloading is another key aspect of client-side performance optimization. Functions like prefetchDNS, preconnect, preload, preloadModule, preinit, and preinitModule, re-exported through packages/react-dom/src/client/ReactDOMClientFB.js and defined in packages/react-dom/src/shared/ReactDOMFloat.js, allow developers to instruct the browser to proactively fetch or prepare critical resources. This can significantly reduce perceived loading times by making assets available before they are explicitly requested.

Form state management utilities provide advanced integration with HTML forms. Hooks such as useFormState and useFormStatus, along with the requestFormReset function, are re-exported via packages/react-dom/src/client/ReactDOMClientFB.js. These utilities facilitate robust and declarative management of form input, submission status, and programmatic resets, streamlining the development of interactive forms within React applications.

Beyond core rendering, React DOM also supports features like createPortal for rendering children into a different part of the DOM tree, outside the parent component's hierarchy, and flushSync for synchronously applying pending React updates. While unstable_batchedUpdates is a legacy no-op in modern React due to automatic batching, it remains present for backward compatibility. The client directory also contains mechanisms for managing default transition indicators, leveraging the browser's Navigation API to display loading states during React transitions, as seen in packages/react-dom/src/client/ReactDOMDefaultTransitionIndicator.js.

For server-side rendering, which complements client-side hydration, refer to React DOM: Server-Side Rendering (SSR) with Fizz and Static Prerendering.

React DOM: Server-Side Rendering (SSR) with Fizz and Static Prerendering

zoom_in

React DOM provides robust server-side rendering (SSR) capabilities, leveraging both modern streaming approaches and legacy synchronous methods to render React components on the server. This functionality supports the initial rendering of web applications, improving perceived performance and SEO. The primary implementations for SSR are found in the packages/react-dom/src/server directory.

The modern approach to SSR in React is based on "Fizz," which enables streaming HTML content progressively to the client. This includes functions such as renderToReadableStream and renderToPipeableStream. renderToReadableStream is designed for environments that support Web ReadableStreams (like browsers, Edge, and Bun), allowing React to render a component tree and stream the resulting HTML to the client as it becomes available. renderToPipeableStream, specific to Node.js, renders a React tree to a Node.js Writable stream, providing similar streaming benefits. These streaming functions integrate with AbortSignal for cancellation, manage stream flow, and handle errors, using shared Fizz server components for request creation and state management.

In addition to initial streaming, React DOM also supports resumable rendering. Functions like resume and resumeToPipeableStream allow a server-side rendering process to continue from a previously captured PostponedState object. This can be beneficial for scenarios where parts of the server rendering might be paused and later resumed, potentially for more efficient resource utilization or complex rendering pipelines.

For static site generation, React DOM offers static prerendering functionalities, including prerender and resumeAndPrerender. prerender initiates static rendering, producing a ReadableStream of HTML and a postponed state that can be used for later resumption. This capability is useful for generating static HTML files that can be served directly, reducing server load and improving page load times for static content. The resumeAndPrerender function then allows for continuing a static prerendering process from a saved state. These static prerendering functions are available across browser, Edge, and Node.js environments.

For backward compatibility, React DOM also includes legacy synchronous rendering methods such as renderToString and renderToStaticMarkup. renderToString renders a React tree to a complete HTML string, while renderToStaticMarkup does the same but omits React-specific attributes, making it suitable for static content where no client-side hydration is needed. These legacy methods do not support features like Suspense.

The entry points for these SSR functionalities are provided for specific environments within the packages/react-dom/src/server and packages/react-dom/npm directories. For instance, packages/react-dom/npm/server.node.js provides Node.js-specific implementations, while packages/react-dom/npm/server.browser.js and packages/react-dom/npm/server.edge.js cater to browser and Edge environments, respectively. React Native Renderer: Event System and Global Responders

zoom_in

The React Native renderer incorporates an event system to translate native events, such as touches or scrolls, into React's synthetic event model. This system manages event flow, ensuring that events are processed and dispatched to the correct React components efficiently. It supports both legacy (Paper) and newer (Fabric) architectures, with specific mechanisms for each.

At its core, the event system uses a plugin architecture to extract and process events. The ReactNativeEventPluginOrder in packages/react-native-renderer/src/ReactNativeEventPluginOrder.js dictates the sequence in which these plugins operate. The system uses SyntheticEvent objects, defined in packages/react-native-renderer/src/legacy-events/SyntheticEvent.js, which normalize native events and are pooled to optimize performance by reducing garbage collection overhead. Event batching, handled by modules like packages/react-native-renderer/src/legacy-events/EventBatching.js and packages/react-native-renderer/src/legacy-events/ReactGenericBatching.js, groups multiple state updates for efficient rendering.

For touch and gesture interactions, the ResponderEventPlugin in packages/react-native-renderer/src/legacy-events/ResponderEventPlugin.js plays a role in determining which component should "respond" to a series of touch events. This plugin manages a global interaction lock and dispatches responder-specific synthetic events like onStartShouldSetResponder and onResponderGrant. It relies on the ResponderTouchHistoryStore in packages/react-native-renderer/src/legacy-events/ResponderTouchHistoryStore.js to track touch history, which is essential for complex gesture recognition.

Event dispatching in the legacy environment is managed by ReactNativeEventEmitter in packages/react-native-renderer/src/ReactNativeEventEmitter.js. This module receives native events and receiveTouches, processing them through _receiveRootNodeIDEvent to resolve the associated React Fiber instance and initiate the event plugin pipeline. For the Fabric environment, ReactFabricEventEmitter in packages/react-native-renderer/src/ReactFabricEventEmitter.js performs a similar role, extracting and batching synthetic events from native events using a plugin system. The ReactNativeBridgeEventPlugin in packages/react-native-renderer/src/ReactNativeBridgeEventPlugin.js translates native events into synthetic events and manages their propagation through capture and bubbling phases.

Global responder handling ensures that only one component at a time can be the active responder for touch events. This is managed by ReactFabricGlobalResponderHandler in packages/react-native-renderer/src/ReactFabricGlobalResponderHandler.js for Fabric and ReactNativeGlobalResponderHandler in packages/react-native-renderer/src/ReactNativeGlobalResponderHandler.js for the legacy system. These handlers communicate with the native UI manager (either nativeFabricUIManager or UIManager) to set or clear the JavaScript responder status on native views. The getListener function in packages/react-native-renderer/src/ReactNativeGetListener.js retrieves event listener functions from a React Fiber's current props, allowing the event system to correctly dispatch events to the component's handlers.

React Server Components (RSC) with Flight and Fizz: Server-side Runtime

zoom_in

React provides server-side rendering (SSR) capabilities through two primary, experimental implementations: Fizz for traditional SSR and Flight for React Server Components (RSC). These components execute entirely on the server, producing a React tree that can be further processed either by the client or through Fizz for additional SSR. This setup allows for streaming and asynchronous operations, enhancing the user experience by delivering content progressively.

The server-side runtime for React Server Components is managed by Flight, whose core functionality is defined in packages/react-server/src/ReactFlightServer.js. Flight orchestrates the rendering and streaming of React trees, handling the lifecycle of render requests from initiation to output. It can serialize various data types, including React elements, server references, Promises, and Iterators, extending the capabilities of structuredClone() for efficient client-server communication. Flight identifies modules marked with "use client" as client references and registers them to encode client-side loading instructions. Conversely, functions or files marked "use server" are treated as server references, acting as server entrypoints callable from the client. Flight also supports prerendering, which optimizes for ahead-of-time rendering by adjusting error handling and streaming behaviors, particularly for error omission in the model.

Server-side action decoding from FormData is handled by packages/react-server/src/ReactFlightActionServer.js, which processes incoming FormData to extract and reconstruct server actions and their associated form state. This mechanism is crucial for executing server functions triggered by client-side interactions. The system uses specific FormData field prefixes to identify server actions and dynamically binds arguments to these server functions.

Fizz, implemented primarily in packages/react-server/src/ReactFizzServer.js, focuses on streaming SSR. It manages the server rendering process, including handling Suspense boundaries to enable progressive content delivery. Fizz processes various React element types, such as function and class components, and integrates server-side React Hooks for managing component state and behavior in the server environment. Hooks like useId and useActionState are specifically tailored for server execution, while client-side specific hooks are generally disabled or provided as no-ops.

The runtime also incorporates robust debugging capabilities, especially within development builds. Mechanisms are in place for detailed component stack tracking in packages/react-server/src/ReactFizzComponentStack.js, providing insights into the component hierarchy for error reporting and suspension debugging. Fizz also manages Thenable objects (Promises) to support suspending and resuming rendering operations, particularly through SuspenseException in packages/react-server/src/ReactFizzThenable.js, which signals to React's internal mechanisms that a component has suspended.

Configuration for both Fizz and Flight requires specific renderer-defined shims to prevent silent failures, as indicated by placeholder files like packages/react-server/src/ReactFizzConfig.js. The npm entry points in packages/react-server/npm dynamically load either development or production bundles based on the NODE_ENV environment variable, optimizing for performance or debugging as appropriate. Furthermore, for Node.js environments, packages/react-server/src/ReactFlightServerConfigDebugNode.js leverages Node.js async_hooks for advanced debugging and causality tracking of asynchronous operations.

React Server Components (RSC) with Webpack: Bundling and Communication

zoom_in

The integration of React Server Components (RSC) with Webpack is managed by the react-server-dom-webpack package, which is responsible for coordinating the bundling, client-server communication, and dynamic module loading necessary for RSCs to function across various JavaScript environments. This package provides mechanisms for both client-side and server-side runtimes, ensuring that React components can be efficiently rendered and interacted with, regardless of whether they execute on the server or the client.

A key aspect of this integration involves the transformation of modules containing 'use client' or 'use server' directives. For Node.js environments, this is handled through Node.js ESM loader hooks provided by ReactFlightWebpackNodeLoader.js. These hooks intercept module loading, parse the module's Abstract Syntax Tree (AST), and apply transformations. Modules marked 'use client' are rewritten such that their exports are wrapped with registerClientReference, ensuring that attempts to invoke client-side code directly on the server result in an error. Conversely, 'use server' modules have their exported functions wrapped with registerServerReference, making them callable from the client. A similar mechanism, packages/react-server-dom-webpack/src/ReactFlightWebpackNodeRegister.js, uses monkey-patching on Module.prototype._compile for CommonJS modules to detect and apply these transformations.

Webpack plays a crucial role in managing the client-side aspects of RSC. The ReactFlightWebpackPlugin in packages/react-server-dom-webpack/src/ReactFlightWebpackPlugin.js identifies client reference modules within the project. It scans for files containing the 'use client' directive and then injects these modules as asynchronous dependencies into a designated client bundle. This process generates client manifests, such as react-client-manifest.json and react-ssr-manifest.json, which map client module identifiers to their corresponding Webpack chunk IDs and export names. These manifests are essential for the React Flight client runtime to dynamically load and hydrate client components as needed.

Communication between the client and server is facilitated through a system of references. ClientReference objects represent client-side modules or functions that might be referenced from the server, while ServerReference objects represent server-side functions callable from the client. These references are tagged with unique symbols (CLIENT_REFERENCE_TAG, SERVER_REFERENCE_TAG) and include metadata like unique IDs and bound arguments ($$bound) for serialization. When a server component refers to a client module, createClientModuleProxy creates a proxy object on the server, allowing the server to interact with the client module's metadata without executing its code. Similarly, when a client invokes a server action, encodeReply serializes the client data, and server-side functions like decodeReply, decodeAction, and decodeFormState (used in packages/react-server-dom-webpack/src/server) deserialize the incoming data, reconstructing server references and executing the appropriate server logic.

The package provides environment-specific client and server bundles within the packages/react-server-dom-webpack/npm directory, with entry points tailored for browser, Node.js, and Edge runtimes. These bundles offer functionalities for hydrating server-rendered content (createFromReadableStream, createFromNodeStream), serializing client-side data for server actions (encodeReply), and managing module loading specific to each environment. For instance, packages/react-server-dom-webpack/src/client/ReactFlightClientConfigBundlerWebpackBrowser.js handles chunk loading in browser environments by patching Webpack's internal loaders, while packages/react-server-dom-webpack/src/client/ReactFlightClientConfigTargetWebpackServer.js supports server-side rendering (SSR) by pre-initializing scripts. The shared directory (packages/react-server-dom-webpack/src/shared) defines common types and constants for import metadata, ensuring a consistent and optimized format for communication between the client and server.

React Scheduler: Cooperative Task Management

zoom_in

The scheduler package provides a cooperative, priority-based task scheduling mechanism, primarily for internal use within React. Its design aims to enhance UI responsiveness by managing and prioritizing tasks, allowing React to break down large units of work into smaller, non-blocking segments.

The scheduler assigns various priority levels to tasks, including NoPriority, ImmediatePriority, UserBlockingPriority, NormalPriority, LowPriority, and IdlePriority defined in packages/scheduler/src/SchedulerPriorities.js. These priorities dictate the order and urgency with which tasks are executed. For efficient management of these prioritized tasks, the scheduler employs a min-heap data structure, implemented in packages/scheduler/src/SchedulerMinHeap.js, which ensures that the highest priority task (or the earliest expiring task) is always at the forefront of the queue.

To adapt to diverse execution environments, the scheduler package provides several environment-specific implementations, often referred to as "forks," located within packages/scheduler/src/forks. The general-purpose scheduler, found in packages/scheduler/src/forks/Scheduler.js, uses browser APIs like MessageChannel or setTimeout for asynchronous execution, ensuring tasks yield control to the main thread to prevent UI blocking. For native environments, such as React Native, packages/scheduler/src/forks/SchedulerNative.js acts as a compatibility layer, leveraging native runtime schedulers when available or falling back to JavaScript implementations. An experimental implementation, packages/scheduler/src/forks/SchedulerPostTask.js, utilizes the browser's scheduler.postTask API for prioritized callback execution. Additionally, a mock scheduler in packages/scheduler/src/forks/SchedulerMock.js is available for testing, offering precise control over time and task execution without reliance on actual browser mechanisms.

The scheduler also incorporates profiling capabilities, defined in packages/scheduler/src/SchedulerProfiling.js. When enabled, this feature records various task and scheduler-related events, such as task start, completion, cancellation, and errors, into an in-memory buffer. This event logging provides detailed insights into task execution and scheduler behavior, aiding in performance analysis and debugging. The behavior of the scheduler, including profiling enablement and yielding thresholds, is further configured through feature flags like those found in packages/scheduler/src/SchedulerFeatureFlags.js and its environment-specific variants.

These utilities handle common tasks such as verifying NPM permissions, interactive user confirmations, executing shell commands, and managing temporary build metadata, contributing to a robust and guided release process.

React Benchmarking Infrastructure

zoom_in

React's benchmarking infrastructure, located in scripts/bench, is designed to systematically measure and analyze the performance of React's rendering engine. It facilitates the comparison of different React buildstypically a local development branch against a remote main branchto identify performance regressions or improvements across various application scenarios.

The process is orchestrated by a runner.js script in scripts/bench/runner.js, which manages the entire benchmarking lifecycle. This includes discovering available benchmarks, building React bundles for both local and remote repositories, executing the benchmarks, and reporting the results. The build process, handled by scripts/bench/build.js, is responsible for cloning or updating Git repositories, checking out specific commits, and compiling React's UMD production bundles. This ensures that benchmarks are run against optimized code.

Individual benchmarks are located in the scripts/bench/benchmarks directory, each focusing on different React application styles and complexities. For instance, scripts/bench/benchmarks/hacker-news simulates a Hacker News-like application, while scripts/bench/benchmarks/pe-class-components and scripts/bench/benchmarks/pe-functional-components benchmark complex Ads Manager-like UIs built with class and functional components, respectively. A unique benchmark in scripts/bench/benchmarks/pe-no-components specifically tests direct React.createElement usage without traditional component abstractions. Each benchmark typically includes a benchmark.js file defining the application UI, a build.js script for preparing its environment by copying React bundles, and an index.html file that serves as the entry point for performance measurement.

The actual performance measurement is performed by benchmark.js in scripts/bench/benchmark.js, which automates the execution of web application benchmarks. This script launches Chrome, often in headless mode, and runs Lighthouse audits multiple times against the benchmark application. It extracts key performance metrics such as user timings and "First Meaningful Paint" data. To ensure robust results, the system employs statistical methods: it collects data across multiple runs, filters outliers using the Median Absolute Deviation (MAD) method, and calculates the mean and standard error of the mean for each metric. The standard error of the mean is estimated using a bootstrap method to provide a measure of variability. A local web server, provided by scripts/bench/server.js, serves these benchmark directories, making them accessible to the Chrome instances during testing.

Finally, the results are processed and displayed by scripts/bench/stats.js. This module generates a comparison table that highlights the percentage change between local and remote benchmark results. It calculates the mean and standard deviation of the ratio of performance metrics using the Delta Method, providing a statistically sound assessment of performance changes with 95% confidence intervals. Improvements are typically colored green, and regressions red, enhancing readability for quick identification of performance shifts. The command-line interface, detailed in scripts/bench/README.md, allows developers to specify which benchmarks to run, whether to skip build steps, and which repository versions to compare.

Jest Test Environment and Utilities

zoom_in

The React project's Jest configurations, located primarily in the scripts/jest directory, establish a consistent and controlled environment for running tests. This infrastructure defines how global variables are configured, integrates custom JSDOM environments, and employs preprocessors for conditional transpilation.

A core aspect of the test environment is the explicit configuration of global variables. The scripts/jest/setupEnvironment.js script dynamically sets critical boolean flags such as __DEV__, __TEST__, __PROFILE__, and __EXPERIMENTAL__ based on NODE_ENV and RELEASE_CHANNEL process environment variables. This enables conditional execution paths within the React codebase, ensuring that tests accurately reflect behavior in development, production, and experimental contexts. Similarly, scripts/jest/setupGlobal.js standardizes the process.env.TZ to 'UTC' for consistent timezone handling across all tests, preventing locale-dependent failures.

The project utilizes specialized JSDOM environments to simulate browser behavior accurately. The base JSDOM environment, defined in scripts/jest/ReactJSDOMEnvironment.js, extends Jest's default JSDOM capabilities. It customizes document.readyState behavior, which is essential for components that depend on the document's loading state. For tests that integrate both client-side React DOM and server-side rendering, scripts/jest/ReactDOMServerIntegrationEnvironment.js provides a hybrid environment. This environment combines NodeEnvironment with ReactJSDOMEnvironment, allowing server-rendered output to be hydrated and interacted with in a browser-like DOM context.

To manage diverse source file types and apply transformations, a preprocessor defined in scripts/jest/preprocessor.js is used. This preprocessor handles various file formats, including CoffeeScript, TypeScript, CSS, and JSON. Crucially, it applies conditional Babel transpilation, adjusting plugins and JSX runtime choices based on the file path, environment (__DEV__), and the ReactVersionTestingAgainst variable. This ensures that the code under test is transformed consistently with how it would be built for different React versions or deployment scenarios, facilitating accurate testing of features and syntax across the project's evolution.

Advanced Jest Testing Features

zoom_in

The React project's Jest testing environment incorporates advanced utilities to ensure robust and consistent test execution. These utilities include custom matchers, mechanisms for conditionally executing tests based on feature flags, assertions for console output, and decoding of production error messages. Strategies are also in place for balancing test execution across parallel runs.

Custom Jest matchers extend the assertion capabilities for React-specific testing scenarios. For instance, the system provides toMatchRenderedOutput via scripts/jest/matchers/reactTestMatchers.js for asserting against the rendered JSX output of React components. This helps verify that components render as expected. Another custom matcher, defined in scripts/jest/matchers/toThrow.js, normalizes error messages across different Node.js versions to ensure consistent error assertion, regardless of variations in V8 engine messaging.

Conditional test execution is managed using feature flags. The getTestFlags function, found in scripts/jest/TestFlags.js, aggregates environment-specific and feature-specific flags. Tests can then use pragmas (e.g., @gate) that reference these flags. The setup in scripts/jest/setupTests.js provides global helpers like _test_gate and gate to evaluate these conditions, allowing tests to run only when relevant features or environment settings are active. If a gated test is expected to fail under certain conditions, expectTestToFail is used to verify that an error indeed occurs.

Assertions for console output are critical for maintaining code quality. The testing setup in scripts/jest/setupTests.js includes mechanisms to patch console methods, ensuring that unexpected console.error, console.warn, or console.log calls during tests result in failures. This helps identify unintended output and ensures that developers are aware of any runtime warnings or errors.

For production builds, React minifies its error messages. The testing infrastructure includes a system for decoding these messages. When NODE_ENV is set to 'production', the ErrorProxy defined in scripts/jest/setupTests.js intercepts error construction. It uses an errorMap (loaded from ../error-codes/codes.json) to decodeErrorMessage and reconstruct the original, unminified message. This allows tests to assert against human-readable error messages, facilitating debugging.

To optimize test execution time, especially in continuous integration environments, the system employs strategies for balancing test runs. The SizeBalancedSequencer, implemented in scripts/jest/sizeBalancedSequencer.js, is a custom Jest test sequencer. It sorts tests by file size and greedily assigns them to the smallest available shard, aiming to distribute the total test workload evenly across parallel execution environments. This reduces the overall time required for the test suite to complete.
